# OpenClaw 个性化新闻系统计划文档——严格评审报告

---

## 一、整体方案优点

**1. 架构完整性高，系统思维成熟**

文档从定位、边界、数据模型、功能模块到执行路线形成闭环。每一层都有明确的职责边界（§2.3 职责矩阵），避免了"什么都想做、什么都做不清"的常见陷阱。特别是将"平台能力复用"作为设计原则之一（第 15.1 节），体现了对 Skill 形态的正确理解。

**2. 七层偏好模型是核心亮点**

相比"关键词 + 黑名单"的简单方案，七层模型对偏好的抽象层次更合理——从主题、来源、形态、深度、判断角度到容忍度和样本，逐层递进、相互正交。这个设计在理论上具备较好的表达能力，冷启动策略（全部 0.5 + 高探索）也是务实的。

**3. 防茧房是内建约束而非附加功能**

这一点设计正确。配额 + 让渡算法（§10.3）有具体数字，热点注入规则（§10.5）有强制通道，反向多样性约束（§10.4）对连续同质化有应对。这比大多数"个性化系统"把茧房问题当补丁强得多。

**4. 去重三层收缩策略设计精细**

§11.1 的"规则归一化 → Jaccard 筛选 → LLM 精确判断"三级漏斗，是务实可行的降成本方案。500 条/天 → 20-50 候选对 → 5-15 次 LLM 调用，这个数字是合理的估算，避开了"每个组合都 LLM"的成本灾难。

**5. 运行保障规则（v2 新增）解决了并发安全问题**

mkdir 原子锁、heartbeat 机制、崩溃恢复、幂等设计（§4.1-4.2）体现了对生产级可靠性的关注。在一个 Skill 方案里考虑这些，不是过度设计——因为 cron 调度本身就会制造并发。

**6. 分阶段路线图清晰，MVP 定义可操作**

Phase 0A/0B/0C 的拆分降低了起步门槛，每阶段有明确的"完成标准"和"集成验证"要求，可测试性强。

## 二、主要风险与缺口

**1. LLM 成本与延迟未做预算评估**

这是整份方案最大的隐患。系统每日需要进行以下 LLM 调用：

- 分类：500 条 × 1 次
- 摘要生成：去重后剩余条数 × 1 次
- 标题近似去重判断：5-15 组 × 1 次（每组 ≤ 10 条批量）
- 事件归并：每日新增事件数 × N 次
- 事件 summary/keywords 生成：每个新事件 × 2 次
- 周报生成：每周一次大调用
- 评分（如果也走 LLM）：未知次数

估算下来 Phase 3 稳定运行后每日 LLM 调用量在 500-1000 次量级。文档完全没有成本估算和延迟预算。Skill 形态下这些调用直接换算成 API 费用，用户在部署前无法做 ROI 判断。

**2. 持久化存储缺乏索引，查询性能存疑**

纯 JSONL + 目录结构解决了"怎么存"，但没有回答"怎么查"。当 `data/news/YYYY-MM-DD.jsonl` 累积到 15000 条之后：
- 按 source_id 过滤需要全表扫描
- 按 categories 交叉查询需要遍历所有文件
- 历史查询（§14.3）的数据定位方法只是描述性的，没有实现路径

这在 Phase 0-2 勉强可接受，但 Phase 3 的"自然语言查询历史"将面临严重性能问题。文档没有任何关于索引或数据库迁移的规划。

**3. 搜索类来源依赖关键词，但关键词生命周期管理空白**

`search` 类型来源（§7.1）的核心依赖是关键词列表，但：
- 关键词多久更新一次？
- 哪些关键词已失效（"Google+ 关闭"之后仍推送相关词）？
- 搜索结果如何去重于已有 RSS/官方来源？

关键词和 RSS 来源之间没有去重机制，很可能制造大量近似重复新闻。

**4. 反事实和纠偏机制停留在文字描述**

§14.2.5 提到"大幅调整 > 0.3 需升级确认"，但：
- 多个 feedback 同时作用时，权重如何叠加？（多个 +0.1 是否等于一次 +0.2？）
- "纠偏"（§10.6）的执行方式只是"保留 ≥ 2% 曝光"，这个 2% 是在配额计算前还是计算后注入的？
- 偏好权重向 0.5 衰减的速率（每 30 天 5%）是否经过验证？

偏好系统的收敛性完全没有数学分析，只有一个"看起来合理"的参数。

**5. 部分 OpenClaw 平台能力映射存疑**

§3.2 的工具映射表看起来整齐，但以下几项值得质疑：

- `cron` 的 `agentTurn` payload 在 isolated session 下能访问工作空间文件吗？上下文继承规则是什么？
- `exec` 执行辅助脚本的能力在 Skill 内可用吗？有没有权限限制？
- "browser" 工具抓取渲染页面的稳定性如何保证？超时、UA、反爬没有讨论
- delivery 的 `bestEffort: true` 模式（快讯）在网络失败时的重试策略是什么？

**6. 事件归并的 LLM Prompt 完全缺失**

§11.1 Step 3 说"执行 LLM 精确归并"，§12.3 枚举了 10 种关系类型，但 LLM 用于判断归并和生成关系类型的 Prompt 模板从未给出。这意味着整个事件归并系统的核心决策点是空白的。

**7. 没有处理内容来源的版权与合规问题**

`community` 和 `ranking` 类型涉及抓取第三方网页（HN、热帖等）。没有提到：
- robots.txt 的尊重策略
- 内容存储的合法性边界（仅存储摘要 vs 存储全文）
- GitHub API 的速率限制和认证方式

**8. 热评置信度衰减方案不明确**

"来源信任"模型中，Source 的 `credibility` 是静态的，但现实是一个来源可能在某段时间可信、之后质量下滑（如媒体被收购、博主转型）。系统没有对来源质量的动态重评估机制。

## 三、可以补充的功能

**1. 在线学习率自适应机制**

当前偏好更新的调整步长（±0.05 ~ ±0.2）是固定值。可以引入基于反馈密度的自适应学习率：用户活跃反馈多时降低步长（精细调优），反馈稀疏时提高步长（快速探索）。这对冷启动和成熟期都适用。

**2. 多语言新闻的翻译桥接策略**

当来源语言与用户偏好不匹配时（如日语 AI 新闻），系统可以选择：
- 展示原文 + 英文标题
- 调用翻译 API 生成中文摘要（增加成本但提升可读性）
- 直接过滤

文档完全没有涉及这个场景。

**3. 新闻"信号质量"评分（独立于 importance_score）**

`importance_score` 衡量新闻本身的重要性，但"信号质量"衡量的是这条新闻是否可靠——同一事件，有一手信源的新闻信号质量高于转发/二手解读。这两个维度可以独立计算，给用户提供额外决策维度。

**4. 来源覆盖率分析面板**

系统可以定期输出"我的兴趣覆盖了哪些主题"、"哪些主题的新闻源太少"的分析，帮助用户主动扩展来源。这与"探索发现"形成闭环。

**5. 离线可读性保障**

对于无网络/部分来源失败的情况，日报应标注"以下来源不可达，本期未纳入"，并提示用户在网络恢复后是否要补采。§4.2 的容错规则只说了"标记"，没有说怎么向用户呈现。

**6. 偏好可视化与手动干预能力**

7 层偏好模型对用户是黑箱。建议增加：
- 偏好雷达图（文本输出即可）
- "一键重置某层偏好"的指令
- 偏好快照对比（本次 vs 上周的偏好变化）

**7. LLM 调用的影子模式**

对于分类、摘要、去重等 LLM 调用，可以增加影子模式：先用 LLM 处理，但结果仅用于内部验证，不影响最终输出。这样可以积累 ground truth 数据，用于后续 prompt 优化和评估。

## 四、可以精简或删除的内容

**1. `full_content` 字段（NewsItem）**

`full_content` 存储原文内容（§5.1），但整个文档中没有一处真正用到这个字段。所有处理逻辑（分类、摘要、去重）都基于 `content_summary`。存储原文在 Phase 0-2 完全是浪费空间，且增加 JSONL 文件大小（每条从 ~1KB 膨胀到 ~10KB 量级），影响读写性能。

**2. `adjust_style` 反馈类型（§14.1）**

"调整风格"作为独立反馈类型存在歧义——它实际上是一组底层参数的打包操作（exploration_appetite、density 等）。拆分成具体参数操作更清晰，也更容易追踪效果。

**3. "专题追踪"作为独立输出类型（§13.2）**

专题追踪本质上是"事件 + 高密度输出 + 较长摘要"，可以用事件时间线（§12）+ 调整输出密度参数实现，不必作为独立输出类型。删除后简化输出层的实现复杂度。

**4. Phase 3 的"研究能力"定义过于模糊**

§16 Phase 3 提到"研究导向总结（阶段性回顾、趋势观察）"，但这个能力与周报（已包含趋势分析）的边界在哪里？文档没有说清楚。Phase 3 实际上是"Phase 2 + 历史查询 + 偏好衰减 + 周报"，"研究能力"是一个容易导致 Scope Creep 的标签。

**5. `media_urls` 字段**

`media_urls` 字段（§5.1 v2 新增）用于存储关联图片/视频 URL，但系统从不消费这个字段——输出模板中没有任何地方引用它。如果未来不打算做"图文预览"或"视频卡片"，这个字段应该删除。

**6. `form_type` 枚举中的部分类型**

`release_note`、`changelog`、`tutorial` 等 form_type 是开发向内容的细分。对于一个通用新闻系统，这些细分类目在分类准确性上的收益可能抵不过分类粒度带来的复杂度。如果以"AI + 科技 + 商业"为主要受众，`analysis`、`news`、`opinion` 三个已足够覆盖绝大多数场景。

## 五、对 OpenClaw Skill 适配性的判断

**适配性总体评级：中等偏上，但存在几个关键卡点。**

### 适配良好的部分

- **Skill 形态选择正确**：SKILL.md + 工作空间文件的组合完全符合 OpenClaw Skill 规范，不需要额外的插件或服务。
- **工具映射合理**：web_fetch、browser、web_search、read/write、message/cron 的组合覆盖了主要功能需求，工具集选择没有硬伤。
- **Cron + isolated session + delivery 的调度架构**是 OpenClaw 的原生能力，集成方式正确。
- **Standing Orders（§3.4）的设计**与 OpenClaw 的 AGENTS.md 集成方式契合。
- **JSONL 存储**不依赖任何外部服务，Skill 自包含性好。

### 适配存疑的部分

**1. 原子锁机制（§4.1.1）是自实现而非平台能力**

mkdir 原子锁是一个聪明的 POSIX trick，但它是"写在 SKILL.md 里的指令"而非 OpenClaw 平台的原生能力。如果未来 OpenClaw 提供原生的 Job 互斥 API（比如 `openclaw job lock`），这套自实现逻辑就会变成冗余代码。更关键的是：isolated session 之间共享文件系统的假设是否在所有部署环境下都成立？（比如多个副本部署在无共享存储的环境中）

**2. SKILL.md 指令的 LLM 可执行性未经验证**

文档定义了 SKILL.md 的结构骨架（§3.1.1 的 9 个部分），但大量处理逻辑以"指令描述"而非"可执行步骤"的形式存在。例如：
- "由 LLM 进行多标签分类"——但分类的 Prompt 模板在哪里？
- "LLM 判断近似重复"——Prompt 是什么？few-shot examples 有吗？
- "事件归并判断"——§11.1 只有文字描述，没有实际 prompt 文本

OpenClaw agent 能否仅凭这些描述性指令稳定执行，取决于 agent 模型对"隐式推理"的依赖程度。这是一个重大风险。

**3. 多步骤流水线的上下文窗口压力**

完整的流水线（§4 流水线图）：采集 → 去重 → 分类/摘要 → 事件归并 → 评分 → 配比 → 输出生成，涉及大量中间状态读写。在 isolated session 下，每次 agent turn 的上下文窗口需要容纳：
- SKILL.md 全文（很长）
- 相关配置文件的读取结果
- 当日/当周新闻数据的部分读取
- 中间处理结果的写入

这些加起来极易超过上下文限制。文档完全没有讨论上下文窗口管理和分批处理策略。

**4. JSONL 作为 Skill 数据格式的操作复杂度**

JSONL 的 append-only 特性适合日志，但不适合随机读写和查询。在 Skill 场景下，这意味着 agent 需要自己实现过滤逻辑（遍历 → 过滤 → 加载），而不是交给数据库处理。对于一个"自然语言交互"的 Skill，每一次"查询某主题的新闻"都触发一次文件扫描，操作体验和响应速度都会打折扣。

**5. `exploration_appetite` 的量化边界模糊**

§5.4 中 `exploration_appetite` 是 0-1 的浮点数，§10.6 说"每 7 天 +0.05，上限 0.4"。但这个参数如何映射到具体的配额比例？在 §10.3 配额算法中，explore 层配额是固定比例（15%/20%），`exploration_appetite` 在哪里被消费？如果这个参数在 Phase 3 之前根本没有实际效果，那它就是一个悬空的配置项。

## 六、10 分制印象分

**6 / 10**

**给分理由：**

方案展现了相当高的设计成熟度和系统性思维，七层偏好模型、三层去重、事件归并等核心功能的设计逻辑清晰，防茧房内建约束的思路正确，v2 的运行保障规则补足了生产级需求的必要考量。

**扣分点（按影响权重）：**

| 扣分项 | 扣分 | 原因 |
|--------|------|------|
| LLM 成本与延迟无预算 | -1.0 | 每日数百到上千次 LLM 调用在 Skill 形态下成本不可忽视，用户部署前无预期 |
| SKILL.md 可执行性不足 | -1.0 | 大量核心逻辑（L prompt、分类、归并）停留在描述性语言，agent 能否稳定执行存疑 |
| 上下文窗口管理缺失 | -1.0 | 完整流水线在 isolated session 下的上下文压力没有讨论，是工程上的重大隐患 |
| 查询性能无规划 | -0.5 | JSONL 无索引，Phase 3 自然语言查询的性能基础不成立 |
| 事件归并 LLM Prompt 空白 | -0.5 | 整个事件系统的核心决策点没有 Prompt 模板 |

**加分项（上限补偿）：**
- 防茧房内建约束设计 +0.3
- 原子锁与崩溃恢复机制 +0.2

**结论：** 这是一份设计思路优秀但工程化程度不足的方案。它更像一份"架构设计文档"而非"可直接转化为 SKILL.md 的规格文档"。在进入 Phase 0 之前，需要将描述性指令全部转化为具体的 Prompt 模板和分批处理策略，并补充 LLM 成本估算。
