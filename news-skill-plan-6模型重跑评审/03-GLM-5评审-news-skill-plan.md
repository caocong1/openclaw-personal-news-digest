# 新闻 Skill 计划文档评审报告

## 1. 整体方案优点

**架构设计成熟**：八层流水线（来源层→分类层→偏好层→去重归并层→时间线层→输出层→反馈学习层→防茧房层）职责清晰，边界明确。Skill 与平台的职责分工表（§2.3）避免了重复造轮子。

**偏好模型有深度**：7 层偏好模型超越了简单的"关键词+黑名单"范式。主题权重、来源信任、信息形态、深度偏好、判断角度、风格容忍度、样本反馈——覆盖了用户兴趣的多维表达。

**防茧房是核心设计约束**：配额分配机制（核心50%/邻近20%/热点15%/探索15%）、反向多样性约束、热点注入规则、偏好纠偏机制均为内建设计，而非可选插件。这是区别于推荐系统茧房问题的关键。

**事件级处理而非链接级**：新闻处理上升到事件维度，时间线追踪能力让用户看到连续发展的故事而非散乱标题。事件生命周期状态机设计合理。

**去重策略分阶段收缩**：链接精确去重→标题近似去重（规则归一化→Jaccard筛选→LLM精判）→事件归并（topic预筛选→关键词匹配→LLM精判），避免了全量 LLM 比对的成本陷阱。

**运行保障考虑周全**：单实例互斥锁（mkdir原子操作）、Run ID、原子写入（tmp+rename）、幂等语义、超时降级、崩溃恢复等机制体现了生产级思维。

**反馈系统设计严谨**：applied.json 账本与 log.jsonl 分离、可重放的衰减协议、append-only撤销机制、全量重建/增量应用双路径，保证了反馈处理的一致性和可追溯性。

## 2. 主要风险与缺口

**LLM 调用成本未量化**：每条新闻需经历分类、摘要、评分、标题去重判断、事件归并判断等多轮 LLM 调用。以 500 条/天估算，日均调用量可能达 1000+ 次，但文档未提供成本预估、预算上限或熔断机制。

**缺少监控告警体系**：无法观测采集成功率、去重命中率、事件归并准确率、偏好漂移速度、输出质量指标。生产环境需要能发现问题而非仅依赖用户反馈。

**Schema 迁移缺乏操作细节**：§6.4 提及 Schema 兼容读取约定，但 Phase 数据迁移描述为"补字段"——缺乏迁移脚本、回滚策略、版本兼容性测试方案。

**并发安全性仅理论设计**：互斥锁设计合理，但未验证 agent 重启、崩溃恢复、并发采集场景下的数据一致性。缺少压力测试计划。

**LLM 输出格式校验不足**：LLM 返回的分类可能不在预定义类目内、摘要可能超长、JSON 可能格式错误——需要 robust 的校验和重试机制。

**定时任务调度细节缺失**：dedup-index 重建、event archive 清理、偏好衰减触发等需要定时任务，但文档仅提及 cron 配置，未说明具体调度策略。

**来源质量评估缺失**：可以添加任意来源，但没有来源内容质量评估、持续可信度追踪、低质量来源自动降级机制。

**反馈异常检测缺失**：恶意/误操作反馈可能破坏偏好模型，缺少反馈合理性校验和异常行为检测。

## 3. 可补充的功能

**成本控制模块**：Token 计数统计、日均/月均成本预算、预算超限熔断开关、低成本模型降级策略（摘要用小模型，分类用大模型）。

**来源健康监控**：采集成功率、日均产出、内容重复率、可信度动态评估、degraded 来源自动告警。

**反馈质量过滤**：单次反馈权重范围校验、短时间大量反馈的冷却机制、反馈效果评估（某次反馈后是否真的改变了输出）。

**操作审计日志**：记录来源配置变更、偏好大幅调整、事件归档等关键操作，便于问题追溯。

**数据备份策略**：明确备份频率、保留时长、恢复流程。当前仅有"自动备份保留最近10个"，缺乏策略文档。

**版本控制与回滚**：SKILL.md 和配置文件的版本标记、变更日志、一键回滚能力。

**测试用例清单**：单元测试（评分公式、去重逻辑）、集成测试（端到端流水线）、回归测试（偏好更新正确性）的具体用例。

**监控仪表盘**：关键指标的可视化展示，便于用户了解系统运行状态和个人偏好演化趋势。

## 4. 可精简或删除的内容

**分类体系过度设计**：12 个顶层类目 + 二级主题 + 标签的三层结构对 MVP 过于复杂。建议 Phase 0-1 仅保留 6-8 个核心类目，标签能力延后。

**输出类型过多**：日报、晚报、快讯、周报、专题追踪 5 种类型。建议 MVP 仅保留日报+快讯，其他延后。

**时间线关系类型过细**：9 种关系类型（initial_report/follow_up/correction/analysis/community_reaction/test_result/patch/comparison/retraction）可简化为 4-5 种核心类型。

**配额让渡规则过度复杂**：§10.3 的四层让渡优先级逻辑可简化为"不足时顺延给下一优先级组"，让 LLM 在生成时做软性调整而非硬性计算。

**重复描述**：§3.2 工具映射表与 §7.1 来源类型表中的工具列有重复；Phase 各阶段"完成标准"与"集成验证"部分重叠。

**输出模板过度规定**：日报/快讯/周报模板可更宽松，让 LLM 根据内容灵活生成，而非严格按格式填空。

## 5. OpenClaw Skill 适配性判断

### 高度适配的方面

| 能力 | 适配性 | 说明 |
|------|--------|------|
| 内容采集 | ✅ 高 | web_fetch / browser / web_search 完整覆盖 RSS/API/网页/搜索 |
| 数据存储 | ✅ 高 | JSONL 按日存储适合 agent 直接读写，无需外部数据库 |
| 调度触发 | ✅ 高 | cron + delivery 配置与输出类型完美匹配 |
| 推送投递 | ✅ 高 | message 工具支持多渠道，无需自建 |
| 运行约束 | ✅ 高 | AGENTS.md 的 Standing Orders 机制可直接承载 |
| 自然语言交互 | ✅ 高 | 来源管理、偏好反馈均通过自然语言完成，是 agent 核心能力 |

### 需要关注的方面

**Skill 文件大小**：当前设计需写入 SKILL.md 的指令非常详细，可能超出上下文窗口的有效利用范围。建议拆分为 SKILL.md（核心指令）+ references/（详细规范）。

**LLM 调用链路长**：单条新闻处理需多轮 LLM，延迟和成本需要权衡。建议引入处理流水线状态机，支持断点续跑。

**状态管理复杂度**：大量配置文件和数据文件需要 agent 在会话间保持状态感知，建议明确"加载-处理-保存"的原子操作边界。

**错误恢复复杂度**：LLM 调用失败、网络超时、数据格式异常等场景需要 agent 具备重试和降级能力，这增加了 Skill 指令的复杂度。

## 6. 印象分

**8.5 / 10**

**加分项**：设计深度超出同类系统平均水平，7 层偏好模型、防茧房机制、事件级处理、时间线能力、反馈系统严谨性均为亮点。v2 迭代说明设计者在持续优化。文档结构清晰，四大部分（基础定义、数据架构、功能模块、执行路线）递进合理。

**减分项**：成本控制缺位（对 LLM 密集型系统是硬伤）、监控告警缺失、Schema 迁移和并发安全性仅有设计无验证、部分功能对 MVP 过度复杂。作为 Skill 形态实施，这些复杂功能可能面临上下文窗口、调用延迟、状态管理的实际挑战。

**总评**：这是一份可直接进入 Phase 0 实施的高质量设计文档，但建议在进入 Phase 1 前补充成本预算机制和核心监控指标。
