# 对《OpenClaw 个性化新闻系统计划文档》的评审报告

## 1. 整体方案优点

### 1.1 结构完整，已经接近“设计规格书”而非想法草稿
这份文档最大的优点是完整度很高，已经覆盖了一个可落地 Skill 所需的关键层面：

- 产品目标与边界
- OpenClaw 平台映射
- 数据模型
- 存储结构
- 核心处理流程
- 反馈闭环
- 分阶段实施路线

这意味着它不是“想做一个新闻 Skill”的概念文，而是有明显工程落地方向的方案文档。

### 1.2 对 OpenClaw 的平台边界理解比较准确
文档明确区分了：

- **Skill 负责策略与编排**
- **OpenClaw 平台负责调度、消息投递、工具执行**

这是正确的。它避免把 Skill 设计成一个伪后端服务，也避免重复造轮子。尤其是把 cron、delivery、message、workspace 文件存储纳入设计，是比较贴合 OpenClaw 使用方式的。

### 1.3 “事件级处理”思路明显优于普通 RSS 聚合
方案没有停留在“抓新闻 → 排序 → 发摘要”的浅层能力，而是进一步设计了：

- 链接去重
- 标题近似去重
- 事件归并
- 时间线跟踪
- 生命周期状态机

这一点很关键。对于新闻类 Skill，真正的体验差异往往不在采集，而在“如何把信息整理成人可消费的连续上下文”。这个方向是对的。

### 1.4 偏好模型设计有野心，而且不是过度简化
7 层偏好模型虽然复杂，但思路成熟，至少意识到了用户偏好不是单轴的。它把以下维度拆开：

- 主题
- 来源信任
- 内容形态
- 深度
- 判断角度
- 风格与容忍度
- 显式反馈样本

这比“关键词订阅 + 黑名单”高级很多，也更符合长期使用的个性化系统。

### 1.5 防茧房机制被当作核心约束，而不是营销口号
很多个性化方案口头上说防茧房，实际没有机制。本文档则给了：

- 配额层
- 让渡规则
- 反向多样性约束
- 热点强制进入
- 周期性 exploration 提升

这说明作者意识到“个性化”和“视野收缩”之间的结构性冲突，并试图在系统级约束里解决，这点是加分项。

### 1.6 工程稳健性设计明显优于常见 Skill 草案
文档补充了不少“真正运行起来才会遇到”的问题：

- 原子锁
- 心跳
- stale lock 接管
- 原子写入
- tmp 清理
- 幂等语义
- partial 状态
- 超时与降级
- append-only 反馈日志
- 全量重建与撤销协议

对 Skill 文档来说，这种运行保障意识是比较难得的，说明不是只考虑 happy path。

### 1.7 Phase 切分合理，避免“一口吃成平台”
从 Phase 0A/0B/0C 到 Phase 3 的拆法是清晰的：

- 先搭框架
- 再验证采集
- 再做日报
- 再引入多来源、偏好、事件、闭环

这个路线基本符合现实中的渐进式构建方式，利于验证 MVP。

## 2. 主要风险 / 缺口

## 2.1 最大风险：方案过于“系统化”，超出了典型 OpenClaw Skill 的舒适区
这份方案非常强，但也有一个明显问题：**它更像一个轻量新闻系统设计，而不只是一个 Skill 设计**。

OpenClaw Skill 擅长的是：

- 指令编排
- 调用平台工具
- 读写少量 workspace 文件
- 完成中低复杂度、可解释、可逐步执行的任务

但方案里有不少内容已经接近“常驻后端逻辑”的范畴：

- 锁竞争与接管
- 批量日志回放
- 两阶段原子提交
- 每日/每周持续演化的数据维护
- 复杂状态机长期一致性
- 大量 JSONL/事件索引/账本协调

这些不是不能做，但会让 Skill 很快变得脆弱、难调试，而且严重依赖 agent 每次都严格按规范执行。

### 2.2 对“LLM 作为稳定数据处理器”的依赖偏高
文档把 LLM 用于：

- 分类
- 摘要
- 标题近似去重精确判断
- 事件归并
- 事件 keywords 生成
- 趋势总结
- 历史查询总结

这在能力上没问题，但在稳定性上有风险：

- 输出格式不稳定
- 同一条新闻在不同运行中分类可能漂移
- 归并判断可重复性差
- keywords 生成容易波动
- 长期回放时结果不一定一致

对于 append-only + 可重放系统，这种不确定性会积累成数据质量问题。

### 2.3 反馈系统设计很完整，但实现成本明显偏高
`applied.json`、`revert_feedback`、`system_decay`、全量重建、原子双提交，这一套设计在工程上是合理的，但对 Skill 来说可能太重了。

风险在于：

- 开发成本高
- prompt 复杂度高
- 调试困难
- 维护门槛高
- 人工审计成本高

如果早期用户量很小、反馈频率不高，这个反馈账本体系可能会“设计过剩”。

### 2.4 去重与事件归并的成本可能低估
文档已经做了收敛设计，但仍有几个风险：

- 标题相似度规则对中英混合、多语言、缩写、别名、二次创作标题可能不稳定
- 同一事件跨不同 topic 时，topic 预筛选会漏合并
- 事件 keywords 只有 3-5 个，可能覆盖不足
- active event 规模增大后，候选筛选仍可能变慢
- 不同来源对同一事件的 framing 完全不同，LLM 归并容易误判

换句话说，这部分是方案的亮点，同时也是主要复杂度黑洞。

### 2.5 “自然语言管理来源/反馈/查询”描述充分，但缺少交互协议细节
文档提到了很多自然语言操作：

- 添加来源
- 删除来源
- 调权重
- 反馈某条新闻
- 查询某事件
- 回溯历史

但缺少更明确的交互约束，例如：

- 输入歧义时默认不执行还是追问？
- 删除来源必须二次确认吗？
- 修改配置是否要写审计日志？
- 查询结果过长时如何分页/摘要？
- 用户说“以后少看这个”，是作用于 topic 还是 source 还是 form？

这些在真实使用中会频繁出现，且直接影响 Skill 可用性。

### 2.6 输出质量目标很高，但缺少“失败时最低可接受输出”的更细规范
文档提到 partial 时只显示标题+来源，但还不够细。缺口包括：

- 如果分类失败但摘要成功，是否允许进入日报？
- 如果 published_at 缺失且摘要失败，是否仍可候选？
- 如果事件归并失败，输出是否回退到逐条新闻？
- 如果一天只有低质量内容，日报是否缩短、还是照常凑满？
- 快讯误报的回退机制是什么？

缺少这些规则时，实际实现容易风格漂移。

### 2.7 计划没有充分考虑 OpenClaw Skill 文档本身的可维护性
文档希望把大量行为都编码进 `SKILL.md`。但如果把以下内容都塞进一个 Skill：

- 采集协议
- 多来源规则
- 分类 prompt
- 去重 prompt
- 归并 prompt
- 输出模板
- 反馈规则
- 查询规则
- 运行保障规则

那么 `SKILL.md` 很可能会变成一个超长、难维护、难调优的巨型指令文件。  
对 Agent 来说，太长的 Skill 也可能降低执行一致性。

## 3. 可以补充的功能

## 3.1 增加“来源健康度面板 / 诊断输出”
现在有来源 stats，但更适合补一个运维视角的功能：

- 最近成功采集时间
- 连续失败次数
- 近 7 天平均产出
- 命中率
- 被去重比例
- 被选入日报比例
- degraded 状态原因

这会极大提升维护性，尤其是来源一多之后。

## 3.2 增加“人工审核/纠偏入口”
对高风险动作应支持人工纠偏，例如：

- 把某条新闻手动归到另一个事件
- 手动解除错误归并
- 手动提升/降低某事件重要性
- 将某来源标记为噪声源
- 对某次反馈做撤销/更正

这类操作未必要一开始全做，但至少应预留数据结构与命令入口。

## 3.3 增加“输出解释性字段”
个性化系统很容易让用户觉得“为什么给我推这个”。建议输出时支持可选的简短解释：

- 因为你近期更关注 AI 工具链
- 因为这是你关注事件的后续
- 因为它属于高重要性公共热点
- 因为这是探索位推荐

这能帮助用户建立信任，也有助于收集反馈。

## 3.4 增加“静默模式 / 低打扰模式”
新闻系统如果长期运行，用户很快会感受到频率压力。可以补充：

- 工作日/周末不同推送策略
- 夜间不推送
- 快讯合并窗口
- 低打扰模式下只推最重要 3-5 条
- 一周无重大内容时自动降频

这类功能和真实体验关系很大。

## 3.5 增加“源类型优先级模板”
来源很多时，用户未必愿意逐一配权。可以提供预设模板：

- 官方优先
- 开发者生态优先
- 快讯型优先
- 深度分析优先
- 中文优先 / 英文优先
- 低噪声模式

这样可减少初期配置成本。

## 3.6 增加“多语言处理策略”
文档有 `language` 字段，但没有把多语言策略讲透。建议补充：

- 输出默认是否统一翻译成中文
- 标题是否保留原文
- 不同语言新闻是否分别去重
- 同一事件跨语种如何归并
- 摘要风格是否依语言调整

对新闻系统来说，这是很现实的需求。

## 3.7 增加“来源白名单 / 黑名单级别的强约束”
目前更偏权重控制，但实际常常需要明确规则：

- 永不采集某来源
- 某来源只允许出现在探索位
- 某来源仅作事件补充，不作为主记录
- 某来源内容必须二次验证后才能进快讯

这能显著提升输出质量。

## 3.8 增加“自检 / 审计任务”
建议增加一个定期审计任务，而不是只做采集和推送：

- dedup-index 是否与最近 7 天文件一致
- active event 是否存在空 item_ids
- feedback log 与 applied.json 是否一致
- 是否存在长期未归档事件
- 是否存在异常大的 output
- 最近 7 天是否出现某来源占比过高

这类巡检功能很适合长期运行系统。

## 4. 可以精简或删除的内容

## 4.1 可以先砍掉“7 层偏好模型”的后半部分
如果目标是先做可用 Skill，建议 MVP/Phase 1 只保留：

- topic_weights
- source_trust
- form_preference
- style.exploration_appetite
- liked/disliked samples

可以先不做或弱化：

- judgment_angles
- depth_preference 的复杂分支逻辑
- tracked_event_types
- blocked_patterns 的复杂模式描述

原因很简单：这些字段价值有，但前期学习成本高、反馈样本不足、实际效果不易验证。

## 4.2 可以先弱化“可重放衰减协议”
`system_decay` 作为 append-only 事件很优雅，但对早期 Skill 来说有点重。初版可以先采用更简单策略：

- 每月一次直接重写 preferences
- 记录 last_decay_at
- 不追求完全回放一致性

等系统成熟再升级到日志化衰减。

## 4.3 可以先不做“revert_feedback + 全量重建”闭环
这个设计从审计角度很漂亮，但不是 MVP 必需。初期可简化为：

- 仅支持追加反馈
- 错误反馈人工直接修 preferences
- 先不做严格的账本一致性体系

否则会把大部分实现复杂度压到一个暂时不高频的功能上。

## 4.4 可以先取消“事件 keywords 自动维护”
事件归并初版可以只依赖：

- topic 预筛选
- 事件标题相似
- LLM 判断

`keywords[]` 是优化项，不一定要在早期引入。否则每次事件更新都要维护额外派生字段，增加不一致风险。

## 4.5 Markdown 表格型时间线输出可以先简化
日报/周报中的时间线不一定要一开始就用表格。表格在某些聊天渠道兼容性一般，也会增加输出噪音。初版更适合：

- 事件标题
- 当前状态摘要
- 最近 2-3 个节点 bullet list

更稳，也更易读。

## 4.6 锁与恢复协议可以适度收缩
对于单用户、单实例、低并发环境，当前锁方案略重。初版可保留：

- 简单目录锁
- 超时接管
- run_id
- tmp 原子写

但可以暂缓：

- 复杂的 meta 损坏分支处理
- 细粒度 heartbeat 协议
- 多种 stale case 的接管逻辑

先保证“够用且可解释”比“极致完备”更重要。

## 5. 对 OpenClaw Skill 适配性的判断

## 5.1 总体判断：**适配，但需要明显收缩为“Skill-first”版本**
这个方案**方向上非常适合 OpenClaw Skill**，因为它的核心确实是：

- 编排
- 规则
- 数据文件持久化
- 定时触发
- 多工具协作
- 消息输出

这些都符合 OpenClaw 的能力模型。

但它当前的问题是：  
**方案设计强度已经部分越过了“Skill 编排”，逼近“小型状态系统”**。  
如果直接原样写成一个 Skill，风险是：

- `SKILL.md` 过长
- 执行不稳定
- 数据规范很细，但 agent 不一定每次都能严格一致执行
- 运维复杂度高于预期

## 5.2 更适合的落地方式
比较合理的 OpenClaw 适配方式是：

### A. Skill 保持“策略与流程定义”
例如：

- 采集流程
- 评分规则
- 输出模板
- 反馈映射
- 查询意图路由
- 风险动作升级条件

### B. 复杂稳定逻辑尽量下沉为辅助脚本
例如可用 `exec` 调用本地脚本处理：

- dedup-index 重建
- JSONL 规范化
- 锁管理
- 反馈日志回放
- 数据审计
- 归档清理

这样可以避免把所有一致性要求都压在 prompt 上。

### C. 分层文档而不是超长单文件
建议不是把一切都写进一个巨大的 `SKILL.md`，而是：

- `SKILL.md`：高层行为编排
- `references/data-models.md`
- `references/output-templates.md`
- `references/feedback-rules.md`
- `scripts/*.py|js`

这种结构更符合长期维护。

## 5.3 结论
**如果把它当成“OpenClaw 内的轻量新闻编排系统”，它是适配的。**  
**如果把它当成“纯靠 SKILL.md 驱动的复杂一致性系统”，适配性会明显下降。**

所以我的判断是：

- **概念适配：高**
- **直接照文档一比一写成单 Skill：中等偏低**
- **经过收缩与分层后再实现：高**

## 6. 10 分制印象分

**8.2 / 10**

### 给分理由
加分项：

- 架构完整
- 产品思路成熟
- 对 OpenClaw 平台理解较到位
- 有明显工程意识
- 不是普通 RSS 聚合方案
- 事件化、时间线、防茧房这三点有辨识度

扣分项：

- 复杂度偏高
- Skill 形态下存在过度设计倾向
- 对 LLM 稳定性的依赖略重
- 一些“漂亮但很重”的机制不适合首版
- `SKILL.md` 可能膨胀成难维护的巨型指令文档

## 总结判断

这是一个**方向正确、完成度高、明显经过认真打磨**的方案。  
它的强项在于：不是“抓新闻”，而是试图做“持续性的个性化新闻编排与观察系统”。这点是有价值的。

但它当前最需要的不是继续加功能，而是**做一次工程收缩**：

- 把真正影响 MVP 体验的能力留下
- 把漂亮但重的机制后置
- 把复杂一致性逻辑从 Skill 文档中剥离
- 把方案从“系统规格书”收束成“适合 OpenClaw Skill 执行的实现计划”

如果这样调整，它会是一个很强的 OpenClaw Skill 方案。
