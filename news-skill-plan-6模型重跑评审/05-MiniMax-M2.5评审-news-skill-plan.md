# OpenClaw 个性化新闻系统计划文档评审报告

## 一、整体方案优点

### 1.1 设计理念先进
- **7 层偏好模型**：远超简单的关键词过滤，覆盖主题、来源、形态、深度、判断角度、风格、样本反馈七个维度，设计理念接近推荐系统的工业级实践
- **防茧房内建约束**：不是事后补救的"附加功能"，而是从配额算法到反向多样性约束的系统级设计，真正平衡个性化与信息多样性
- **事件级处理思路**：不仅做链接去重，还上升到事件归并和时间线追踪，符合新闻消费的实际需求

### 1.2 技术实现周全
- **三层去重策略**：URL 精确去重 → Jaccard 标题近似 → LLM 精确判断，层层过滤且计算成本可控
- **运行保障规则**：原子锁、Run ID、幂等设计、崩溃恢复、容错降级，考虑了实际运行中的各种边界情况
- **数据模型清晰**：每个实体（NewsItem、Event、Source、UserPreference、FeedbackEntry）都有详细的 JSON Schema 定义

### 1.3 平台集成合理
- 充分利用 OpenClaw 原生工具（`web_fetch`、`browser`、`web_search`、`message`、cron job）
- Skill 职责边界清晰：逻辑层归 Skill，调度推送层归平台
- Standing Orders 设计合理，平衡了自动化与人工审批

### 1.4 演进路线明确
- Phase 0 拆分为 0A/0B/0C 三个子阶段，可执行性高
- 每个阶段有明确的交付物和验证标准

## 二、主要风险/缺口

### 2.1 LLM 依赖成本风险
- 分类、摘要、去重判断、事件归并、关键词生成均依赖 LLM 调用
- 文档未提及：
  - LLM 调用失败时的降级策略（已提及但较简略）
  - 批量处理时的 token 成本估算
  - 是否考虑使用更轻量的模型处理简单任务
- **风险**：高频运行（如每2小时快讯）可能导致 LLM 调用成本失控

### 2.2 存储性能隐患
- 使用 JSONL 作为主要存储格式（每日一个文件）
- 问题：
  - 30 天累积后单文件可能很大（~15MB/日）
  - 每次评分排序需要全量读取当日数据
  - dedup-index 维护复杂度随时时间增长
- **建议**：考虑引入索引文件或轻量级数据库（如 SQLite）

### 2.3 文档过长导致实施困难
- 文档超过 1500 行，过于详尽
- 问题：
  - 核心逻辑与细枝末节混在一起
  - 部分读者可能抓不住重点
  - 后续维护成本高
- **建议**：拆分为架构文档 + 实现手册 + 操作手册

### 2.4 缺少明确的量化指标
- 评分公式有参数，但未说明：
  - 各参数的经验值来源
  - 如何 A/B 测试调优
  - 如何观测偏好模型是否在向正确方向收敛
- **缺口**：上线后的效果评估和迭代机制

### 2.5 跨境/多语言支持考虑不足
- 分类体系以中文语境设计（游戏、科学、宏观政策）
- 来源配置示例偏少
- 跨语言新闻的处理（如翻译需求）未涉及

## 三、可以补充的功能

### 3.1 性能优化相关
- **缓存机制**：热门来源增量拉取（If-Modified-Since、Etag）
- **分页/限流**：避免单次拉取过多内容导致超时
- **预计算**：每日评分结果预计算，避免每次输出都重新排序

### 3.2 用户体验相关
- **摘要改写**：允许用户要求"用更直白的语言解释"
- **阅读进度**：记录用户已读/未读，避免重复推送
- **噪声过滤**：可配置的"低质量内容"自动过滤（如纯广告、机器人生成的无效内容）

### 3.3 可观测性相关
- **统计面板**：来源采集成功率、LLM 调用次数、用户反馈采纳率
- **异常告警**：连续失败、降级状态时的通知
- **A/B 测试框架**：支持对评分公式、配色方案的对照实验

### 3.4 高级功能（可选）
- **多用户支持**：同一 Skill 服务多个用户，共享来源配置但独立偏好
- **分享功能**：生成的日报/周报可生成分享链接
- **API 暴露**：允许其他系统查询新闻库

## 五、对 OpenClaw Skill 适配性的判断

### 5.1 高度适配的部分
- **工具链映射**：每个功能都能找到对应的 OpenClaw 工具（`web_fetch`、`browser`、`web_search`、`message`）
- **Cron Job 集成**：日报/快讯/周报的触发和推送利用 cron + delivery，设计合理
- **Skill 职责边界**：编排逻辑在 SKILL.md，调度推送在平台层，符合 Skill 的设计哲学

### 5.2 需要适配的部分
- **原子锁实现**：
  - `mkdir` 作为锁原语在 OpenClaw 环境中可能受限（工作空间是否为真实文件系统？）
  - 建议改用工作空间文件锁（如 `data/.lock` 文件存在性判断）
- **LLM 调用次数**：
  - 密集的 LLM 调用（分类、摘要、去重、归并、评分）可能超出 Skill 的预期负载
  - 建议明确每个步骤的调用成本，或考虑引入轻量模型
- **长时间运行**：
  - 采集 + 处理 + 生成一个完整日报可能需要 10-15 分钟
  - 需要确认 OpenClaw 单次任务超时限制

### 5.3 需要验证的假设
- JSONL 文件读写在 OpenClaw 工作空间中是否支持并发安全
- cron job 的 `delivery` 配置是否如文档所述完整可用
- Skill 是否可以访问上一次运行的上下文（如 dedup-index）

### 5.4 整体判断
**适配性评级：B+（8/10）**

核心思路与 OpenClaw Skill 模型高度契合，大部分设计可直接迁移。需重点验证文件锁实现和 LLM 成本两个关键点。
