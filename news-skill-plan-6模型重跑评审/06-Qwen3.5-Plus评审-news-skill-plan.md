# OpenClaw 新闻 Skill 计划文档评审报告

## 1. 整体方案优点

### 架构设计层面
- **Skill 形态定位准确**：明确将项目定义为 OpenClaw Skill 而非独立后端服务，充分利用平台原生工具链（`web_fetch`、`browser`、`web_search`、`message`、`cron`），避免重复造轮子。
- **分层架构清晰**：来源层→分类层→偏好层→去重归并层→时间线层→输出层→反馈学习层→防茧房层，各层职责边界明确，便于分阶段实现和调试。
- **数据模型完备**：NewsItem、Event、Source、UserPreference、FeedbackEntry 五大核心模型覆盖全业务流程，字段设计细致且预留扩展空间（`_schema_v` 版本控制）。

### 核心机制亮点
- **7 层偏好模型**：远超简单的"关键词 + 黑名单"设计，涵盖主题权重、来源信任、形态偏好、深度偏好、判断角度、风格容忍度、样本反馈七个维度，能精细刻画用户兴趣。
- **防茧房内建约束**：将防茧房设计为核心约束而非附加功能，通过配额分配（核心 50%/邻近 20%/热点 15%/探索 15%）和反向多样性约束确保信息多样性。
- **事件级归并思维**：从链接级去重上升到事件级归并，配合时间线追踪（9 种关系类型），能呈现新闻的连续发展脉络而非散乱标题堆砌。
- **运行保障完善**：v2 新增的原子锁机制（`mkdir` 原子操作）、Run ID 追踪、崩溃恢复、幂等语义、超时降级等规则，体现生产级系统思维。

### 工程实践优秀
- **Schema 版本控制**：每条记录含 `_schema_v`，支持不同版本共存，读取时缺失字段用默认值推导，避免破坏性迁移。
- **原子写入协议**：所有数据写入采用"临时文件 + rename"模式，配合 Run ID 追踪，确保崩溃后数据一致性。
- **反馈可追溯**：Feedback log 追加写入不可修改，配合 `applied.json` 独立账本管理已应用状态，支持撤销补偿（`revert_feedback`）和全量重建。

## 2. 主要风险/缺口

### 技术实现风险
- **LLM 调用成本与延迟**：分类、摘要、去重、归并、评分、时间线更新、周报生成等环节均依赖 LLM，日均 500 条新闻可能产生数百次 LLM 调用，需评估成本和经济性。
- **标题近似去重的误判率**：Jaccard 相似度 + LLM 两阶段收缩虽能减少调用次数，但跨语言、跨文化语境下的标题语义相似性判断仍存在误判风险（尤其是翻译新闻）。
- **事件归并的边界模糊**：同一事件的不同角度报道（如"产品发布"vs"市场反应"vs"技术分析"）是否应归并为同一事件，LLM 判断标准难以统一。
- **原子锁的并发边界**：`mkdir` 原子锁适用于单节点，但 OpenClaw 若支持多节点并发执行，需考虑分布式锁机制（文档未覆盖此场景）。

### 数据治理缺口
- **隐私与合规未涉及**：用户偏好数据（尤其是反馈样本）包含个人兴趣画像，文档未提及数据加密、访问控制、导出/删除机制等合规要求。
- **来源可信度评估主观**：`Source.credibility` 和 `source_trust` 依赖人工配置或用户反馈，缺乏客观的可信度评估框架（如事实核查记录、历史准确率追踪）。
- **冷启动周期过长**：完全依赖反馈学习收敛偏好，初期输出质量可能不稳定，文档未设计加速冷启动的机制（如初始兴趣问卷、示例新闻选择）。

### 运维监控缺失
- **可观测性不足**：缺少系统健康指标定义（如采集成功率、去重命中率、偏好收敛度、配额达成率），难以定位性能瓶颈或异常。
- **告警机制空白**：连续采集失败、LLM 调用异常、数据文件损坏等场景未定义告警条件和升级路径。
- **备份恢复细节模糊**：虽提及"保留最近 10 个备份"，但未说明备份触发条件、存储位置、恢复验证流程。

## 3. 可以补充的功能

### 用户体验增强
- **偏好可视化面板**：提供用户查看当前偏好状态的能力（如"你的 AI 主题权重为 0.7，高于平均"），增加透明度和可控感。
- **手动纠偏入口**：允许用户一键重置偏好至冷启动状态，或手动调整某类目权重（配合 Standing Orders 的升级确认规则）。
- **新闻标记收藏**：支持用户将新闻标记为"稍后阅读"或"收藏"，与反馈系统解耦，满足临时保存需求。

### 智能化升级
- **来源质量自动评估**：基于历史采集数据（如用户反馈命中率、去重淘汰率）自动调整来源权重和可信度，减少人工配置负担。
- **相似新闻推荐**：在日报/周报中增加"如果你喜欢这条，可能也会关注"的关联推荐，利用事件归并和时间线数据。
- **趋势预测能力**：基于事件发展速度和来源覆盖度，预测某事件的热度走势（上升/平稳/衰退），在周报中呈现。

### 协作与分享
- **偏好模板分享**：允许用户导出/导入偏好配置文件，便于同兴趣群体共享配置或跨设备迁移。
- **日报转发优化**：支持将单条新闻或整个日报以格式化消息转发至其他渠道，附带来源链接和摘要。

## 4. 可以精简或删除的内容

### 过度设计的部分
- **Phase 0 三子阶段拆分过细**：Phase 0A（骨架搭建）、0B（单来源采集）、0C（日报生成）可合并为单一 Phase 0，三者依赖紧密且交付周期短，拆分增加管理开销。
- **FeedbackEntry 的 `feedback_id` 生成规则复杂**：`"fb-" + SHA256(timestamp+type+target_id)[:12]` 可简化为时间戳 + 随机数，无需密码学强度。
- **9 种时间线关系类型冗余**：`follow_up`、`test_result`、`patch`、`comparison` 可合并为`update`，由 LLM 生成 `brief` 描述具体进展，减少枚举维护成本。

### 可延后的功能
- **偏好衰减机制**：30 天衰减周期和 Catch-up 算法实现复杂，初期可简化为手动触发或固定周期执行，待系统稳定后再优化。
- **全量重建路径**：增量应用 + 全量重建的双路径设计适用于高频反馈场景，但 MVP 阶段反馈量有限，可仅实现增量路径。
- **Embedding 替代 Jaccard 的升级路线**：文档提及"候选对 > 100 或 Jaccard FP 率 > 20%"时切换 Embedding，但 MVP 阶段可直接使用 LLM 判断，避免引入额外依赖。

### 文档冗余内容
- **重复的 [v2 新增] 标记**：同一章节内多次出现"[v2 新增]"，可在章节开头统一说明 v2 变更摘要，正文中无需逐处标记。
- **Cron Job 配置示例过细**：三个 cron 配置示例可精简为一个通用模板，具体参数由用户按需调整，无需在 Skill 文档中固化。

## 5. 对 OpenClaw skill 适配性的判断

### 高度适配的方面
- **工具链复用**：文档规划的采集（`web_fetch`/`browser`/`web_search`）、存储（`read`/`write`）、推送（`message`）、调度（`cron`）均对应 OpenClaw 原生工具，无需扩展平台能力。
- **Skill 指令驱动**：SKILL.md 正文结构（角色定义、工作空间说明、采集/处理/评分/输出/反馈/查询指令、运行约束）符合 AgentSkills 规范，agent 可按指令直接执行。
- **工作空间数据持久化**：配置和数据文件存储于 `<skill-workspace>` 目录下，利用 OpenClaw 工作空间机制实现持久化，无需外部数据库。
- **自然语言交互**：来源管理、偏好调整、历史查询等能力通过自然语言与 agent 交互，符合 OpenClaw 的对话式操作范式。

### 需确认的平台能力
- **Cron Job 的 `delivery` 配置**：文档假设 cron job 支持 `delivery` 字段将输出投递到聊天渠道，需确认 OpenClaw 是否原生支持此功能或需额外配置。
- **原子文件操作的跨平台一致性**：`mkdir` 原子锁、`rename` 原子写入在 Linux/macOS 上可靠，但需确认 OpenClaw 运行环境是否统一为 POSIX 兼容系统。
- **LLM 批量调用支持**：标题近似去重和事件归并需批量 LLM 判断（每组≤10 条），需确认 OpenClaw 是否支持批量 prompt 或需多次独立调用。

### 适配性结论
**该方案与 OpenClaw Skill 模型高度适配**。核心设计原则（规则与策略由 Skill 承载、编排逻辑由 SKILL.md 定义、数据持久化于工作空间、利用平台工具执行）完全符合 AgentSkills 规范。主要风险在于部分功能（如 cron delivery、批量 LLM 调用）需确认平台支持程度，但不影响整体可行性。

## 6. 10 分制印象分

### **综合评分：8.5 / 10**

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | 9/10 | 分层清晰、职责明确、扩展性强 |
| 数据模型 | 9/10 | 五大模型覆盖全面，版本控制到位 |
| 工程实践 | 9/10 | 原子写入、崩溃恢复、幂等语义专业 |
| 用户体验 | 8/10 | 7 层偏好精细，但冷启动和可视化待增强 |
| 可维护性 | 8/10 | Schema 版本和反馈追溯优秀，监控告警缺失 |
| 平台适配 | 9/10 | 工具链复用充分，少量能力需确认 |
| 文档质量 | 8/10 | 结构完整，但 v2 标记冗余、部分细节过细 |

### 加分项
- 防茧房机制内建为核心约束，体现系统思维深度
- 反馈可追溯设计（log + applied.json + revert 补偿）达到审计级标准
- 运行保障规则（原子锁、Run ID、超时降级）展现生产级考量

### 减分项
- 缺少可观测性和告警机制定义
- 隐私合规和数据来源合法性未涉及
- 部分功能过度设计（如 Phase 0 拆分、9 种时间线关系）

---

**总体评价**：这是一份成熟的、生产级的 Skill 设计方案，在个性化、防茧房、事件追踪等核心能力上展现出深入思考。主要改进空间在于简化 MVP 范围、补充运维监控能力、确认平台依赖细节。如按计划实施，有望成为 OpenClaw 生态中的标杆 Skill。
