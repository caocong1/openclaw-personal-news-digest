# OpenClaw 个性化新闻系统计划文档评审报告

## 1. 整体方案优点

### 1.1 架构设计成熟
- **7层偏好模型**设计精细，涵盖主题、来源、形态、深度、判断角度、风格容忍度、样本反馈，远超简单的关键词过滤方案
- **三层去重策略**（链接级→标题近似→事件归并）层次分明，兼顾效率与准确性
- **防茧房机制内建化**，通过配额分配、反向多样性约束、热点注入等形成系统性保障，而非附加功能

### 1.2 数据模型完善
- `NewsItem`、`Event`、`Source`、`UserPreference`、`FeedbackEntry` 五大核心实体定义清晰
- Schema 版本控制 (`_schema_v`) 和字段兼容性约定支持平滑演进
- JSONL 追加写入 + 原子 rename 的写入策略符合数据完整性要求

### 1.3 工程实践考虑周全
- **单实例互斥锁**采用原子 mkdir + heartbeat 机制，避免并发冲突
- **幂等设计**区分 exact/near duplicate 两种语义，支持断点续跑
- **崩溃恢复**机制完整：tmp 文件清理、stale 锁检测、双文件原子提交

### 1.4 与 OpenClaw 平台深度集成
- 明确区分 Skill 职责与平台能力边界，避免重复造轮子
- Standing Orders 定义清晰的授权范围和升级条件
- Cron job 配置示例可直接使用

## 2. 主要风险/缺口

### 2.1 性能与成本风险
| 风险点 | 说明 | 严重程度 |
|--------|------|---------|
| LLM 调用量估算不足 | 标题去重阶段 C 每日 5-15 次、事件归并 Step 3 每篇新闻 1-5 次，500条/天 ≈ 500-1000 次 LLM 调用/天 | 中高 |
| 无 embedding 降级方案 | v2 将 embedding 方案推迟到 Phase 2+，但 Jaccard 在大规模时效率存疑 | 中 |
| 无缓存策略 | 相同/相似新闻的 LLM 分类/摘要结果未设计缓存复用 | 中 |

### 2.2 数据一致性风险
- **反馈撤销机制**虽设计完善，但全量重建路径在反馈量大时（如运行数月后）性能开销显著
- `applied.json` 账本与 `preferences.json` 双文件原子提交，但无校验和机制检测静默损坏

### 2.3 冷启动与收敛风险
- 冷启动所有主题权重 0.5，前几次输出"偏向均匀分布"，但缺乏明确的新用户引导策略
- `exploration_appetite` 初始 0.3 与上限 0.4 差距过小，探索空间受限

### 2.4 来源可靠性风险
- 未设计来源健康度自动降级后的**自动恢复**机制（需人工干预）
- 无来源内容质量评分（如点击率、用户反馈聚合），仅靠 `consecutive_failures` 判断

### 2.5 缺失能力
- **无多语言处理策略**：采集多语言新闻后的翻译/统一展示方案未提及
- **无图片/视频处理**：`media_urls` 字段存在但无处理逻辑
- **无用户画像导出/迁移**：长期用户偏好数据无法导出或迁移

## 3. 可以补充的功能

### 3.1 高优先级
| 功能 | 说明 |
|------|------|
| LLM 结果缓存层 | 对相同 URL/title 的摘要、分类结果缓存 7 天，减少重复调用 |
| 来源质量评分 | 基于点击率、用户反馈、采集成功率综合计算来源质量 |
| 多语言统一处理 | 采集新闻语言检测 → 翻译为系统语言 → 原文保留链接 |
| 用户画像快照 | 支持导出/导入 preferences.json + feedback log，便于迁移 |

### 3.2 中优先级
| 功能 | 说明 |
|------|------|
| 事件重要性预测 | 基于早期信号（来源权威性、传播速度）预测事件发展潜力 |
| 个性化摘要长度 | 根据 `depth_preference` 动态调整摘要生成 prompt |
| 周报自动生成摘要 | 当前周报模板含"趋势观察"，但无具体生成方法 |
| 来源自动发现 | 基于用户高评分新闻反向发现新来源（需人工确认） |

### 3.3 低优先级
| 功能 | 说明 |
|------|------|
| 新闻情感分析 | 为每条新闻附加情感标签（正面/负面/中性） |
| 阅读时间估计 | 基于 `word_count` 估算阅读时长 |
| 热点预测 | 基于传播速度预测即将成为热点的新闻 |

## 5. 对 OpenClaw Skill 适配性的判断

### 5.1 适配优势
- ✅ **形态契合**：Skill = 指令文档 + 工作空间数据，本方案完全匹配
- ✅ **工具链复用**：`web_fetch`、`browser`、`web_search`、`read/write`、`message` 等工具使用合理
- ✅ **调度集成**：Cron job + delivery 配置与 OpenClaw 原生能力无缝衔接
- ✅ **渐进交付**：Phase 0-3 分阶段演进，符合 Skill 迭代开发模式

### 5.2 潜在挑战
| 挑战 | 说明 | 缓解建议 |
|------|------|---------|
| 复杂状态机 | Event 生命周期、偏好更新等状态机较复杂，Agent 理解成本高 | 提供状态流转图 + 关键决策点 checklist |
| LLM 调用频繁 | 每日数百次 LLM 调用，对 token 消耗较大 | Phase 0 即引入缓存层，监控调用量 |
| 数据量增长 | 30 天 15MB 虽可控，但长期运行后 feedback log 膨胀 | 设计 log 归档策略，保留最近 90 天明细 |
| 多 Agent 并发 | OpenClaw 可能多实例运行，锁机制需充分测试 | Phase 0B 增加并发测试用例 |

### 5.3 适配建议
- **SKILL.md 结构**：严格遵循 §3.1.1 的 9 部分结构，每部分以"步骤编号 + 条件判断"形式编写
- **Prompt 模板外置**：将 LLM 分类/摘要/去重 prompt 提取到 `config/prompts/` 目录，便于调优
- **调试接口**：增加 `dry-run` 模式，采集后不写数据仅输出日志，便于开发调试

| 架构完整性 | 9/10 | 7层偏好、三层去重、防茧房机制设计精良 |

| MVP 可行性 | 8/10 | Phase 0 拆分合理，可快速验证核心流程 |

### 总分：7.8/10

**一句话评价**：这是一个设计精良、考虑周全的新闻推荐系统方案，工程实现难度适中，与 OpenClaw Skill 形态高度契合，建议在 Phase 0 即引入 LLM 缓存层并精简文档篇幅后推进实施。
