  <files>references/scoring-formula.md, references/processing-instructions.md</files>
  <action>
**In references/scoring-formula.md**, add a new section "## Provenance Modifier (Post-Formula)" after the existing "## Phase Activation History" section (or at the end if that section does not exist). This section documents:

1. A provenance modifier table:
   - T0: 1.15 (rare, highest-value original content)
   - T1: 1.10 (direct source)
   - T2: 1.05 (original reporting)
   - T3: 1.00 (neutral)
   - T4 with no direct-coverage sibling in same event: 1.00
   - T4 with T0/T1/T2 sibling in same event: 0.75

2. The `adjusted_score = final_score * provenance_modifier` formula, applied AFTER the 7-dimension formula computes `final_score`. This is a post-formula multiplier -- it does NOT add an 8th dimension or change any existing weights.

3. The lookup pseudocode for `lookup_provenance_modifier(item, provenance_db, events)`:
   - Join path: `NewsItem.id -> data/provenance/provenance-db.json` to get `ProvenanceRecord.tier`
   - For T4 items: check `Event.item_ids` from `data/events/active.json` for sibling items with tier T0/T1/T2
   - If no provenance record exists for the item, modifier = 1.00 (neutral)
   - If item has no event_id, modifier for T4 = 1.00 (no same-event context)

4. A note that these modifier values are configurable starting points and may be tuned after deployment observation.

**In references/processing-instructions.md**, add two changes:

A. In Section 4 Scoring (around line 964), after step 3 ("Sort all scored items by final_score descending"), add a new step 4:
```
4. Apply provenance modifier: For each scored item, compute `adjusted_score = final_score * lookup_provenance_modifier(item)` using the provenance modifier table from `references/scoring-formula.md`. Re-sort by `adjusted_score` descending. All downstream steps (repetition penalty, representative selection, quota allocation) use `adjusted_score` instead of `final_score`.
```

B. Add a new "## Section 4R: Event Representative Selection (PIPE-03)" between Section 4A (cross-digest repetition penalty) and the quota allocation steps. This section contains:

1. Trigger: Runs AFTER Section 4A repetition penalty, BEFORE Section 4 quota allocation (Step 1-3).
2. Purpose: For events with multiple items in the scored pool, select exactly one representative item per event. Non-representative items are excluded from the digest scoring pool for this run only.
3. Algorithm:
   - Group scored items by `event_id` (items with null event_id are always eligible, skip them)
   - For groups with only 1 item, that item is the representative (no action)
   - For groups with 2+ items, sort by: tier rank ascending (T0=0, T1=1, T2=2, T3=3, T4=4), then source credibility descending (from `config/sources.json`), then `adjusted_score` descending
   - First item after sort = representative. Set `Event.representative_item_id = item.id` in `data/events/active.json`
   - All other items in the group are excluded from the digest pool for this run (set runtime flag `digest_pool_excluded: true` -- do NOT persist this flag to the JSONL file, do NOT modify the persistent `digest_eligible` field)
4. The TIER_RANK map: `{ "T0": 0, "T1": 1, "T2": 2, "T3": 3, "T4": 4 }`
5. Explicit warning: "Non-representative exclusion is runtime-only. Do NOT write `digest_eligible: false` or any persistent exclusion flag back to the news JSONL file."

Also update the SKILL.md Output Phase step 1 description to mention provenance modifier application and representative selection between scoring and quota allocation. Add after the existing scoring sentence: "Apply provenance modifiers from `references/scoring-formula.md` to compute `adjusted_score`. Run event representative selection per Section 4R before quota allocation."
