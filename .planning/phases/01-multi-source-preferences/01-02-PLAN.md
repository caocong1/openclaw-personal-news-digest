**## Overview**
Brief description: This document specifies how user feedback is processed into preference updates. It is referenced by SKILL.md "User Commands" section when the agent detects feedback intent.

**## Feedback Type Mapping** (FB-01)
Table with columns: Type | Example Phrases | Target Field | Adjustment | Notes

| Type | Example Phrases | Target Field | Adjustment | Notes |
|------|----------------|--------------|------------|-------|
| more | "more AI news", "show more about tech" | topic_weights[matched_topic] | +0.1 | Match topic from user text against categories.json IDs |
| less | "less gaming", "fewer finance stories" | topic_weights[matched_topic] | -0.1 | Clamp to [0.0, 1.0] |
| trust_source | "this source is great", "I trust 36Kr" | source_trust[source_id] | +0.15 | Match source by name/ID from sources.json |
| distrust_source | "downgrade this source", "less from X" | source_trust[source_id] | -0.2 | Clamp to [-1.0, 1.0] |
| like | reply "good", "like #3" | feedback_samples.liked_items | append item ref + topic +0.05 | Micro-adjust: related topic_weight +0.05 |
| dislike | reply "bad", "dislike #5" | feedback_samples.disliked_items | append item ref + topic -0.05 | Micro-adjust: related topic_weight -0.05 |
| block_pattern | "no clickbait", "block rumor" | form_preference[matched_form] + blocked_patterns | form -0.2, append pattern | Record pattern string for future filtering |
| adjust_style | "more exploration", "dense format" | style.* | direct set | Map: "more exploration" -> exploration_appetite +0.1, "dense" -> density "high" |

**## Feedback Reference Disambiguation** (FB-03)
When user gives feedback, resolve which item/source/topic they mean using this cascade:
1. **Message reply context**: If the user replies to a digest message, the replied message provides context
2. **Sequence number**: If user says "#3" or "item 3", map to the 3rd item in the most recent digest
3. **Keyword search**: Search recent items (last 24h JSONL) by title keyword match
4. **Source name match**: Match against sources.json by name (case-insensitive substring)
5. **Topic match**: Match against categories.json category IDs
6. **Ambiguous**: If no match or multiple matches, list top 3 candidates and ask user to clarify

**## Incremental Preference Update Procedure** (FB-02)
Step-by-step processing flow:
1. Read `config/preferences.json`
2. Check `feedback_processing_enabled`. If `false`, skip all processing and log "feedback processing disabled" (FB-04 Kill Switch)
3. Read `data/feedback/log.jsonl`
4. Filter entries where `timestamp > preferences.last_updated` (if last_updated is null, process all entries)
5. Sort filtered entries by `timestamp` ascending (oldest first)
6. **Per-session cumulative cap**: Track cumulative change per topic_weight per run. If any single weight's cumulative change would exceed +/- 0.3, stop applying further changes to that weight and log warning (prevents feedback loop runaway per Research Pitfall 3)
7. For each entry:
   a. Look up adjustment in Feedback Type Mapping table
   b. **Large adjustment check** (FB-05): If this single adjustment would change any preference value by > 0.3, DO NOT apply. Instead, log the pending change and escalate to user: "Feedback would change [field] by [amount]. Confirm? (yes/no)". Mark entry as `status: "pending_confirmation"` in log.
   c. Apply adjustment, clamp values to valid ranges (topic_weights: [0.0, 1.0], source_trust: [-1.0, 1.0], form_preference: [-1.0, 1.0])
   d. Increment `total_feedback_count`
8. **Backup** (PREF-05): Before writing updated preferences, copy current `config/preferences.json` to `data/feedback/backup/preferences-{ISO8601_timestamp}.json`. List all backup files sorted by name, delete oldest if count > 10.
9. Update `last_updated` to current ISO8601 timestamp
10. Atomic write updated `config/preferences.json` (write to `.tmp.{run_id}`, rename)

**## Kill Switch** (FB-04)
- Field: `preferences.feedback_processing_enabled` (boolean)
- When `false`: Skip step 3-10 entirely. Log "Kill switch active: feedback processing disabled."
- The kill switch does NOT prevent feedback LOGGING -- user feedback is still appended to log.jsonl. It only prevents preference UPDATES.
- To re-enable: User sets `feedback_processing_enabled: true` (or agent sets it on user command)

**## Escalation Thresholds** (FB-05)
- Single adjustment > 0.3: Escalate to user confirmation before applying
- Per-session cumulative cap: Max +/- 0.3 net change to any single topic_weight per pipeline run
- These thresholds align with SKILL.md Standing Orders "Large preference weight change" escalation condition

**## Preference Backup Management** (PREF-05)
- Backup location: `data/feedback/backup/`
- Naming: `preferences-{YYYY-MM-DDTHH:mm:ss}Z.json`
- Retention: 10 most recent backups. Delete oldest when count exceeds 10.
- Backup is created BEFORE each preference update, not after.

**feedback_boost computation (PREF-01, PREF-03):**

```
feedback_boost calculation:
  Read config/preferences.json -> feedback_samples, source_trust

  Initialize boost = 0

  1. Category match (liked): If item.primary_category matches any liked_item's category -> boost += 0.3
  2. Source trust: If item.source_id exists in source_trust AND source_trust[source_id] > 0 -> boost += min(source_trust[source_id], 0.2)
  3. Category match (disliked): If item.primary_category matches any disliked_item's category -> boost -= 0.3
  4. Source distrust: If item.source_id exists in source_trust AND source_trust[source_id] < 0 -> boost -= min(abs(source_trust[source_id]), 0.2)
  5. Blocked pattern: If item matches any blocked_pattern (substring match on title or tags) -> boost -= 0.5
  6. Clamp result to [0.0, 1.0]

  If feedback_samples is empty (no liked/disliked items, no trusted/distrusted sources):
    feedback_boost = 0 (cold start, same as Phase 0)
```
