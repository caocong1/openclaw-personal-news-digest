**## Circuit-Breaker Enforcement** (COST-02) -- Insert BEFORE the existing batch processing section, since the circuit-breaker check must happen before any LLM call.

```
Before each LLM batch:
1. Read config/budget.json
2. Compute usage_ratio = calls_today / daily_llm_call_limit
3. Also compute token_ratio = tokens_today / daily_token_limit
4. Use the HIGHER of the two ratios as effective_usage

If effective_usage >= 1.0 (100%):
  CIRCUIT BREAK
  - Stop ALL LLM processing (classify, summarize, filter, extract)
  - EXCEPTION: Daily digest output generation is exempt (1 final LLM call allowed for digest assembly)
  - Leave remaining items as processing_status: "raw" (breakpoint resume will pick them up next run)
  - Log to metrics: "circuit_breaker_activated: true"
  - Add to digest footer: "Warning: LLM budget exhausted. Some items unprocessed."

If effective_usage >= alert_threshold (default 0.8):
  WARNING
  - Log warning: "LLM budget at {percentage}%"
  - CONTINUE processing normally (warning only, do not stop)
  - Add to digest footer: "Note: LLM budget at {percentage}%"
  - This triggers SKILL.md Standing Orders escalation: notify user about approaching limit

If effective_usage < alert_threshold:
  PROCEED normally
```

**## LLM Result Cache** (COST-03) -- Insert AFTER circuit-breaker, BEFORE batch processing. This is the cache-before-call pattern.

```
Cache files:
- data/cache/classify-cache.json -- keyed by URL SHA, stores classification results
- data/cache/summary-cache.json -- keyed by URL SHA, stores summary results

Cache Lookup (before each LLM batch):
1. For each item in the batch:
   a. Compute url_sha = SHA256(normalized_url)[:16] (same hash as dedup-index)
   b. Look up url_sha in classify-cache.json (or summary-cache.json for summarize step)
   c. If found AND (now - cached_at) < 7 days:
      - Apply cached result to item (set categories, importance_score, form_type, tags for classify; set summary for summarize)
      - Remove item from the LLM batch (skip the call)
      - Increment run metrics: cache_hits += 1
   d. If found but cached_at > 7 days:
      - Delete stale entry
      - Keep item in batch (needs fresh LLM call)
   e. If not found:
      - Keep item in batch

Cache Write (after each LLM batch):
1. For each item that received a fresh LLM result:
   a. Write to cache: url_sha -> { result fields, cached_at: now ISO8601 }
2. Atomic write the cache file (tmp + rename)

Cache Cleanup (at start of each pipeline run):
1. Read classify-cache.json and summary-cache.json
2. Delete all entries where (now - cached_at) > 7 days
3. Atomic write cleaned cache files
4. Log: "Cache cleanup: removed {N} stale entries from classify-cache, {M} from summary-cache"
```

**## Tiered Model Strategy** (COST-04) -- Insert after cache section.

```
Model tier guidance for LLM calls:

Fast model (lower cost, sufficient for structured tasks):
- Classification (classify.md prompt) -- structured JSON output, well-defined categories
- Summarization (summarize.md prompt) -- short factual summaries
- Search result filtering (filter-search.md prompt) -- binary keep/discard decisions
- Content extraction (extract-content.md prompt) -- structured data extraction

Strong model (higher cost, needed for nuanced reasoning):
- Event merging and timeline analysis (Phase 2)
- Weekly report generation with cross-domain synthesis (Phase 3)
- Ambiguous feedback interpretation when disambiguation cascade fails
- Complex query answering from history

Implementation note: If the OpenClaw platform supports model selection via prompt parameters or system settings, specify the tier. If not, use the default model for all tasks and log this as a future optimization opportunity. The tiered guidance is documented here so it can be activated when platform support is confirmed.
```
