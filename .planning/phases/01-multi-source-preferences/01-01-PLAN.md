**1. Create `references/prompts/filter-search.md`** -- LLM prompt template for filtering web_search results.
- Input: array of search results (title, url, snippet) + the source's filter context (what topics the user cares about)
- Task: discard irrelevant results (product ads, forum posts, old content), keep genuine news/analysis
- Output: Return ONLY a valid JSON array of kept items with fields: title, url, snippet
- Include instruction to discard results that appear older than 48 hours based on visible date cues
- Follow the same JSON-only output convention as classify.md and summarize.md

**2. Create `references/prompts/extract-content.md`** -- LLM prompt for extracting structured data from rendered web pages.
- Input: rendered page text (from web_fetch or browser) + extraction_type parameter ("official_announcement" | "community_posts" | "ranking_list")
- Task: extract individual news/content items from the page
- Output: Return ONLY a valid JSON array of extracted items with fields: title, url (absolute), snippet (max 500 chars)
- Include per-type extraction hints (ranking: look for numbered lists; community: look for post titles + links; official: look for dated announcements)

**3. Extend `references/collection-instructions.md`** -- Add 5 new sections AFTER the existing RSS section. Each section follows the same structure: When to use, Source config example, Fetch steps, Error handling, then feeds into the shared normalize-dedup-write pipeline.

Sections to add:
- **## GitHub Release/Repo Collection** (SRC-02): `web_fetch` the API URL (`https://api.github.com/repos/{owner}/{repo}/releases`) with `extractMode: "text"`. Parse JSON array. Map fields: `name` -> title, `html_url` -> url, `body` (truncated to 500 chars) -> content_snippet, `published_at` -> published_at. fetch_config carries `owner`, `repo`, `endpoint` (releases|commits), `per_page` (default 10). Rate limit note: 60 req/hr unauthenticated, optional `token` field for 5000 req/hr. On 403/rate-limit, increment consecutive_failures and skip.

- **## Search-Based Collection** (SRC-03): For each keyword in `fetch_config.keywords`, call `web_search(keyword)`. Collect all result URLs and titles. Pass batch to LLM with `references/prompts/filter-search.md`. LLM returns filtered JSON array. Follow shared normalize-dedup-write pipeline. fetch_config carries `keywords` (string array), `max_results` (default 10). Include date-range keywords in searches (append current year) for freshness.

- **## Official Announcement Collection** (SRC-04): `web_fetch` the source URL with `extractMode: "text"`. If result is too short or appears JavaScript-dependent, fall back to `browser(url)`. Pass rendered text to LLM with `references/prompts/extract-content.md` (extraction_type: "official_announcement"). fetch_config carries `prefer_browser` (boolean, default false). Follow shared pipeline.

- **## Community Page Collection** (SRC-05): Use `browser(url)` to render the page (community pages typically require JS). Pass rendered text to LLM with `references/prompts/extract-content.md` (extraction_type: "community_posts"). If browser fails, log error, increment consecutive_failures, skip source for this run. fetch_config carries `max_items` (default 15).

- **## Hot Ranking Collection** (SRC-06): Try `web_fetch` first. If result appears incomplete, fall back to `browser(url)`. Pass to LLM with `references/prompts/extract-content.md` (extraction_type: "ranking_list"). fetch_config carries `prefer_browser` (boolean, default false), `max_items` (default 20).

After all source-type sections, add two new sections:

- **## Source Management Commands** (SRC-07, SRC-10): Document natural language command handling. When user intent is detected as source management:
  - **Add source**: Parse user's description (e.g., "add LangChain GitHub releases"). Infer type from keywords (GitHub/release -> github, search/find/monitor -> search, official/blog/announcement -> official, community/forum/reddit -> community, ranking/trending/hot -> ranking, RSS/feed/atom -> rss). Construct source config with sensible defaults (weight 1.0, credibility 0.5 for new sources). Ask user to confirm before writing to sources.json.
  - **Delete source**: Match by name or ID. ALWAYS require second confirmation per Standing Orders ("Delete source 'X'? This cannot be undone. Confirm yes/no").
  - **Enable/Disable**: Toggle the `enabled` field. Confirm action.
  - **Adjust weight**: Accept relative ("increase weight of X") or absolute ("set X weight to 0.5"). Clamp to [0.1, 2.0].
  - **Disambiguation** (SRC-10): If user input is ambiguous (e.g., "add Apple" could mean Apple Inc news or Apple developer blog), list 2-3 candidate interpretations and ask user to pick. If a source name closely matches an existing source (edit distance <= 2 or substring match), warn about potential duplicate.

- **## Source Health Metrics Computation** (SRC-08): After each pipeline run (after processing, before output), compute stats for each source that contributed items in this run:
  - `selection_rate` = items_selected_for_output / total_fetched (last 7 days of JSONL data)
  - `dedup_rate` = items_deduped / total_fetched (last 7 days)
  - `fetch_success_rate` = successful_fetches / total_fetch_attempts (last 7 days, from metrics files)
  - `quality_score` = selection_rate * 0.4 + (1 - dedup_rate) * 0.3 + fetch_success_rate * 0.3
  - If `total_fetched < 7` for a source, keep all rates at 0.5 (insufficient data).
  - Write updated stats back to sources.json atomically.
