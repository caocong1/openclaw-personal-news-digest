1. In `references/processing-instructions.md`, replace the current "Section Assignment" subsection within Section 4 (Output Generation) with a new `### Quota-Based Section Assignment (ANTI-01, ANTI-02)` subsection. The current approximate text ("Core Focus ~50% of items...") is replaced by the formal algorithm from 02-RESEARCH.md "Pattern 4: Anti-Echo-Chamber Quota Algorithm":

   **Step 1 -- Classify items into quota groups:**
   - Read config/preferences.json topic_weights and config/categories.json adjacent mappings
   - "core": item.categories.primary has topic_weight >= 0.7
   - "adjacent": item.categories.primary is in the adjacent list of any core category (and not itself core)
   - "hotspot": item.importance_score >= 0.8 AND not core AND not adjacent
   - "explore": everything else
   - **Cold-start handling:** When NO topic reaches topic_weight >= 0.7, use the top-3 topics by weight as pseudo-core. This preserves quota structure during cold start.

   **Step 2 -- Compute target counts:**
   - core_target = round(N * 0.50), adjacent_target = round(N * 0.20), hotspot_target = round(N * 0.15), explore_target = round(N * 0.15)
   - Where N = target item count (15-25 for daily digest)

   **Step 3 -- Fill from each group (top-K by final_score):**
   - Select top items from each group up to target

   **Step 4 -- One-way chain yielding (ANTI-02):**
   - If explore underfilled: yield remaining slots to adjacent
   - If adjacent underfilled: yield remaining slots to hotspot
   - If hotspot underfilled: yield remaining slots to core
   - Direction is one-way only: explore -> adjacent -> hotspot -> core (never reverse)

   **Step 5 -- Reverse diversity constraints (ANTI-03):**
   - Read last 3 days of data/metrics/daily-*.json to get category_proportions history
   - If same topic > 60% of digest for 3 consecutive days: cap that topic at 50% today (replace excess with next-best items from other categories)
   - If same source > 30% of digest for 3 consecutive days: cap that source at 20% today
   - If same event pushed > 3 consecutive days: include only if new developments (relation != "initial")
   - **Grace period:** If fewer than 3 days of metrics with category_proportions data exist, skip ANTI-03 constraints entirely. Log "ANTI-03 grace period: insufficient history."

   **Step 6 -- Hotspot injection (ANTI-04):**
   - After quota fill, check for items with importance >= 0.8 that were excluded
   - Force-inject into candidate pool (add to the selected list)
   - Still subject to title dedup (no duplicates) and quality checks

   **Step 7 -- Preference correction (ANTI-05):**
   - After final selection, check if each of 12 categories has >= 2% exposure in the final list. If any category has 0% and items exist for it, swap the lowest-scored item in the largest group with the highest-scored item from the missing category.
   - Check exploration_appetite auto-increase: read preferences.json style.exploration_appetite and last_exploration_increase date. If >= 7 days since last increase: set exploration_appetite += 0.05 (cap at 0.4), update last_exploration_increase to today. Write atomically.

   **Step 8 -- Tag items with quota group:**
   - Each selected item gets a `quota_group` tag: "core" / "adjacent" / "hotspot" / "explore"
   - Used by output template to assign to correct section

2. Update `references/data-models.md` DailyMetrics schema to add fields needed for ANTI-03:

   - `source_proportions`: fraction of selected items per source (for ANTI-03 source concentration check)

   Add to DailyMetrics field notes: "quota_distribution, category_proportions, and source_proportions are written during output generation. Used by ANTI-03 reverse diversity constraints to check 3-day history."
