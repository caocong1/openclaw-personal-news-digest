Create SKILL.md as the main orchestration entry point. It MUST be under 3000 tokens (~2000 Chinese characters). Use `{baseDir}` as the workspace root reference throughout.

**YAML frontmatter** (from RESEARCH.md Pattern 1):

```yaml
---
name: news-digest

```

**Section 1: Role** (2-3 sentences)
- You are a news research assistant running in the OpenClaw workspace
- Working directory: {baseDir}
- Core task: collect, process, score, and deliver personalized news digests

**Section 2: Collection Phase** (numbered steps, brief)
1. Acquire file lock (read data/.lock, check age, write if absent/stale, skip if locked < 15 min)
2. Generate run_id: run-YYYYMMDD-HHmmss-XXXX
3. Read config/sources.json for active sources
4. For each RSS source: web_fetch with extractMode "text", extract title/link/description/pubDate
5. For each item: normalize URL (strip utm_* params, https, www, trailing slash, lowercase host), compute SHA256[:16]
6. Check data/news/dedup-index.json -- skip if hash exists
7. Write new items to data/news/YYYY-MM-DD.jsonl (use atomic write: .tmp.{run_id} then rename)
8. Update dedup-index.json atomically

**Section 3: Processing Phase** (numbered steps, brief)
1. Read references/prompts/classify.md for classification prompt template
2. Read references/prompts/summarize.md for summary prompt template
3. Collect items with processing_status "raw" from today's JSONL
4. Batch 5-10 items per LLM call for classification (categories, importance, form_type, tags)
5. Batch 5-10 items per LLM call for summarization (2-3 sentence Chinese summary)
6. On LLM failure: retry once; if still fails mark processing_status "partial"
7. On classify fail but summarize success: mark item for exploration slot
8. Update budget.json counters (check current_date, reset if new day, then increment)
9. Write updated items back to JSONL atomically

**Section 4: Output Phase** (numbered steps, brief)
1. Read references/scoring-formula.md for scoring rules
2. Score all completed items, sort by final_score descending
3. Read references/output-templates.md for digest format
4. Apply quality gate: if fewer than 3 items, output shortened version; if 0 items, skip output entirely
5. Generate daily digest markdown, write to output/latest-digest.md atomically
6. Write daily metrics to data/metrics/daily-YYYY-MM-DD.json

**Section 5: Standing Orders** (from gpt-plan-v3.md Section 3.4)
Authorization scope:
- Execute news collection and output generation on cron trigger
- Auto-execute dedup
- Auto-update budget counters

Escalation conditions (require human confirmation):
- Delete source configuration
- Large preference weight change (single change > 0.3)
- Daily LLM calls approaching budget limit (> 80%)

Prohibitions:
- Do not add sources without user confirmation
- Do not send preference data to external services
- Do not generate empty output when no content exists
- Do not exceed daily LLM budget for non-essential calls

**Section 6: Operational Rules** (brief, ~3-4 items)
- File lock: acquire-or-skip, 15 min expiry (reference Section 2 step 1)
- Atomic writes: all data files via tmp + rename pattern
- Schema versioning: all JSON records include _schema_v, read with missing-field defaults
- Crash recovery: on startup, delete data/**/*.tmp.* files older than 15 minutes

**CRITICAL: Token budget.** After writing, verify the content is under 3000 tokens. If it exceeds, trim verbose descriptions -- keep only step numbers and brief instructions, move details to comments or references. The executor MUST count tokens and iterate until under budget.
