Create the title near-duplicate detection pipeline and multi-language processing support.

1. Create `references/prompts/dedup.md` -- the LLM prompt template for batch title near-duplicate judgment. Use the exact template from 02-RESEARCH.md "Dedup Prompt Template" section:
   - Input: `{title_list_with_ids}` placeholder for batch of candidate title pairs
   - Judgment criteria: same event + same angle = duplicate; same event + different angle = NOT duplicate; related topic = NOT duplicate
   - Output: JSON array of duplicate groups (arrays of IDs), or empty array if no duplicates
   - Language: Chinese (consistent with classify.md and summarize.md prompts)

2. Add new `## Section 1A: Title Near-Duplicate Detection (PROC-04)` to `references/processing-instructions.md`, inserted between current Section 1 (Batch LLM Processing) and Section 2 (Error Handling). This section describes the 3-stage funnel:

   **Stage A -- Rule Normalization:**
   - Strip common prefixes/suffixes: "Breaking:", "Update:", "[Video]", "[Exclusive]", "快讯:", "独家:", "【视频】", etc.
   - Remove all punctuation, collapse whitespace, lowercase
   - Preserve original title for display (normalization is for comparison only)
   - Set `language` field on each item: detect by checking if majority of characters are CJK (U+4E00-U+9FFF) -> "zh", otherwise "en"

   **Stage B -- Jaccard Bigram Similarity:**
   - For each pair of items in the same day's batch with the SAME `language` value (NEVER compare cross-language):
     - Generate character bigrams from normalized title
     - Compute Jaccard similarity = |intersection| / |union|
     - If Jaccard >= 0.6: add to candidate group
   - Efficiency: only compare items sharing the same primary category OR same source_id
   - Expected: ~500 items/day -> 20-50 candidate pairs

   **Stage C -- LLM Precise Judgment:**
   - Only for candidate groups from Stage B (not all items)
   - Batch up to 10 candidate titles per LLM call using `references/prompts/dedup.md`
   - Parse LLM response: array of duplicate group ID lists
   - For each duplicate group: keep the item with highest source credibility as primary
   - Mark secondary items: set `dedup_status: "title_dup"`, set `duplicate_of` to primary item's id
   - Expected: 5-15 LLM calls/day for this step

   **Cross-language rule (PROC-06):** Explicitly state that Stage B MUST skip pairs where `item_a.language != item_b.language`. Cross-language merging happens only at event level in Plan 02.

3. Add new `## Section 1B: Multi-Language Processing (PROC-06)` to processing-instructions.md, immediately after Section 1A:
   - Language detection: CJK character ratio check (if >50% CJK characters -> "zh", else "en")
   - Title dedup: per-language independent pipelines (as described in Stage B above)
   - Summarization: always output Chinese summary regardless of source language (existing behavior in summarize.md)
   - Output display format for English items: `### Original English Title (Chinese Translation)\n{Chinese summary}\nSource: ... | ...`
   - Event merging: cross-language allowed (handled in Plan 02)

4. Update `references/data-models.md` NewsItem schema:
   - `dedup_status` enum: change from `"unique|url_dup"` to `"unique|url_dup|title_dup|event_merged"` (add title_dup and event_merged values)
   - `language` field: already present as `"zh|en"` in schema -- add field note: "Detected during title dedup Stage A. CJK majority -> zh, otherwise en."
   - `duplicate_of` field note: update to mention it is set for both url_dup and title_dup cases
   - Bump `_schema_v` to 3 in the example, add defaults for new fields at bottom: `dedup_status` default "unique" for older records
