<task type="auto">
  <name>Task 1: Expand health-check.sh with alert conditions and weekly inspection, add AlertCondition schema</name>
  <files>scripts/health-check.sh, references/data-models.md</files>
  <action>
1. Rewrite `scripts/health-check.sh` to support two modes: daily quick check (existing behavior, expanded with MON-02 alerts) and weekly full inspection (MON-03). Mode is selected by a `--mode` flag: `bash health-check.sh [base_dir] [--mode daily|weekly]`. Default mode is "daily".

**Daily mode (MON-02 alert conditions) -- expand existing checks and add:**

Keep all 6 existing checks (dedup-index JSON validity, budget date, stale lock, orphaned temp, today's JSONL, latest digest). Then add:

   a. **All-source failure alert**: Read last 2 days of `data/metrics/daily-*.json`. If `sources.failed == sources.total` for both days: print `ALERT: All sources failed for 2 consecutive days`.
   b. **Budget alert**: Read `config/budget.json`. If `calls_today / daily_llm_call_limit >= 0.80`: print `ALERT: LLM budget at {pct}%`. If >= 1.0: print `ALERT: LLM budget EXHAUSTED -- circuit breaker active`.
   c. **Dedup-index consistency**: For each entry in `data/news/dedup-index.json`, check if a matching JSONL record exists in the last 7 days of JSONL files. Count orphaned entries (in dedup-index but not in any JSONL). If orphaned > 10% of total: print `ALERT: Dedup-index has {N} orphaned entries ({pct}% of total)`.
   d. **Source concentration alert**: Read today's metrics. If any single source accounts for > 50% of items_fetched: print `ALERT: Source concentration -- {source_id} accounts for {pct}% of items`.
   e. **Empty digest alert**: Read today's metrics. If `output.generated == false` and `items.fetched > 0`: print `ALERT: Items fetched but no digest generated`.

   At the end of daily mode, write alert summary to stdout. Alerts use the prefix `ALERT:` for easy grep filtering. Non-alert findings use `WARN:`, `INFO:`, or `OK:` prefixes (existing pattern).

**Weekly mode (MON-03 full inspection) -- run daily checks PLUS:**

   f. **Dedup-index rebuild check**: Count entries in dedup-index.json. Count entries in last 7 days of JSONL files. If difference > 20%: print `WARN: Dedup-index drift -- index has {N} entries, JSONL has {M} unique URLs. Consider rebuilding.`
   g. **Empty events check**: Read `data/events/active.json`. Flag events with `item_ids` length == 0 (should not exist).
   h. **Long-stable events**: Flag events with status "stable" and last_updated > 14 days ago (candidates for manual review or forced archival).
   i. **Source success rates**: For each source in `config/sources.json`, compute success rate from last 7 days of metrics. Flag sources with < 50% success rate.
   j. **Preference extreme values**: Read `config/preferences.json`. Flag any topic_weight > 0.95 or < 0.05 (may indicate runaway feedback).
   k. **Cache size check**: Count entries in `data/cache/classify-cache.json` and `data/cache/summary-cache.json`. If either > 5000 entries: print `WARN: Cache large -- {file} has {N} entries. TTL cleanup may be needed.`

   Keep the script using python3 for JSON parsing (consistent with existing pattern and Phase 0 decision).

2. Add `## AlertCondition` to `references/data-models.md` after DailyMetrics:

```json
{
  "_schema_v": 1,
  "type": "source_failure|budget_warning|budget_exhausted|dedup_inconsistency|source_concentration|empty_digest",
  "severity": "warning|critical",
  "message": "string (human-readable alert message)",
  "details": {},
  "detected_at": "ISO8601"
}
```

   Field notes: AlertCondition objects are output to stdout by health-check.sh. They can optionally be collected into daily metrics under an `alerts` array field. Add `"alerts": []` to DailyMetrics schema.

1. Rewrite `scripts/data-archive.sh` to implement per-data-type TTL rules (MON-04):

   Keep existing structure (BASE_DIR parameter, echo-based logging) and expand:

   a. **News JSONL (30-day TTL)**: Keep existing logic -- delete JSONL files older than 30 days. Already implemented.
   b. **Dedup-index entries (7-day TTL)**: Read `data/news/dedup-index.json`. For each entry, check `fetched_at`. Remove entries where fetched_at is older than 7 days. Write cleaned index atomically (write to .tmp, then mv). Log count of removed entries.
   c. **Feedback detail (90-day TTL)**: Read `data/feedback/log.jsonl`. Filter out entries where `timestamp` is older than 90 days. Write remaining entries to tmp file, then rename. Log count of archived feedback entries.
   d. **Cache files (7-day TTL)**: Read `data/cache/classify-cache.json` and `data/cache/summary-cache.json`. Remove entries where `cached_at` is older than 7 days. Write atomically. Log counts. (This supplements the per-run cache cleanup in processing-instructions.md Section 0B with a batch cleanup.)
   e. **Metrics files (30-day TTL)**: Keep existing logic -- already implemented.
   f. **Orphaned temp files (15-min TTL)**: Keep existing logic -- already implemented.
   g. **Archived events**: No TTL -- permanent storage. Log count of archived event files for awareness.

   Maintain the python3 JSON parsing approach. Use atomic writes (tmp + mv) for all JSON modifications. Print summary at end with counts per data type.
